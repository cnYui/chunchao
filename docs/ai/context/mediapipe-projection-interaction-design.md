# MediaPipe 投影交互方案设计

## 背景

- 本文把用户口中的 `Media Pipeline` 按 `MediaPipe` 理解
- 当前页面已经具备完整的乐器斥力表现层，核心代码在 `main.js`
- 现阶段缺的不是动效，而是把“真实手部靠近”映射成页面里的交互输入

## 真正要解决的问题

- 目标不是单纯识别一只手
- 目标是把“墙上的投影乐器”和“真实空间里的手”映射到同一个交互坐标系
- 如果只判断手在二维画面里接近某个乐器区域，单机前视摄像头即可完成
- 如果要判断手是否真的靠近墙面上的乐器，必须补充一个深度通道

## 当前页面可复用部分

- 乐器锚点、大小和排斥参数已经稳定
- 斥力逻辑本质上只依赖一个接近源：
  - 是否激活
  - 当前坐标
  - 可选强度
- 这意味着当前鼠标输入可以被抽象成通用交互输入，不需要推翻现有页面结构

## 推荐的阶段拆分

### 第一阶段：前视单机 MVP

- 设备：投影电脑 + 自带摄像头
- 识别：浏览器端使用 MediaPipe Hand Landmarker
- 输入：左右手都可作为交互源，但同一时刻只选一只主交互手
- 映射：使用掌心中心点映射到页面二维坐标
- 触发：掌心进入某个乐器的影响半径时，触发该乐器斥力

这一阶段的目标不是解决真实空间深度，而是先验证：

- 投影环境下摄像头能否稳定识别手
- 左右手切换是否自然
- 乐器斥力节奏是否符合装置体验

### 第二阶段：前视 + 侧视双机位

- 设备：
  - 电脑前视摄像头：负责 `x / y`、左右手、置信度
  - 手机侧拍摄像头：负责估算手到墙面的距离
- 结论：手机虽然没有深度传感器，但依然可以成为“深度通道”
- 原因：这里需要的是“到墙面的相对距离”，不是高精度三维重建

## 为什么前视单机不能解决“真实靠近”

- MediaPipe Hand Landmarker 输出的 `landmarks.z` 是相对深度，原点在手腕附近
- `world landmarks` 的原点在手的几何中心，不是墙面
- 所以它们都不能直接回答“手距离投影墙面还有多远”
- 前视摄像头可以判断“手在画面里靠近了乐器区域”，但不能稳定判断“手已经贴近墙面”

## 双机位方案的最小实现

### 前视通道

- 使用 MediaPipe Hand Landmarker
- 输出：
  - `handedness`
  - 掌心中心 `x / y`
  - 手部检测置信度
  - 时间戳

### 侧视通道

- 手机侧拍固定在能同时看到墙面和平行于墙面的手部活动区域的位置
- 先做一次简化标定，确定：
  - 侧视画面中的墙面线
  - 活动区域的有效范围
- 每帧只求一个值：手的代表点到墙面线的距离
- 这个距离经过归一化后作为 `depthToWall`

### 融合结果

- 两路数据最终收敛成统一输入结构

```ts
type InteractionSource = {
  active: boolean;
  x: number;
  y: number;
  depthToWall: number | null;
  handedness: "Left" | "Right" | "Unknown";
  confidence: number;
  timestamp: number;
};
```

- 第一阶段没有侧视通道时，`depthToWall` 为空
- 第二阶段增加规则：
  - 只有 `depthToWall` 小于阈值时，才允许该手驱动乐器斥力

## 为什么先用掌心，不先用指尖

- 指尖抖动更明显
- 投影装置里高亮区域和阴影会加重抖动
- 掌心更适合作为“接近源中心”，后续若要做拨动、点按，再引入指尖

## 输入层建议

- 不要让 `main.js` 继续直接依赖鼠标事件
- 应拆成两层：
  - 输入采集层：鼠标 / MediaPipe / 手机侧拍
  - 场景响应层：根据统一输入计算乐器斥力

这样做的原因：

- 鼠标可以继续保留成调试后备输入
- MediaPipe 接入不会污染斥力逻辑
- 后续加手机通道时不需要重写乐器动画部分

## 通信建议

- 第一阶段：全部在投影电脑浏览器内完成，不需要额外通信
- 第二阶段：手机侧拍数据通过局域网发送到投影电脑页面
- 实现上不能假设静态浏览器页面自己充当 WebSocket 服务端
- 可选方案：
  - 增加一个很小的本地 relay 服务，用 WebSocket 做转发
  - 或者使用 WebRTC DataChannel 直连手机与投影电脑
- 不建议第二阶段还坚持纯静态无通信
- 原因：手机如果参与实时估距，就必须把数据送回渲染主机

## 风险

- 投影画面会反照到用户手部，可能降低前视摄像头对手的识别稳定性
- 如果前视摄像头和投影光轴太接近，手遮挡投影时会造成更明显的识别抖动
- 手机侧拍如果没有固定机位，所有深度估算都会失效
- 双机位时间戳不同步时，快速挥手会出现“位置到了但深度还没到”的错位

## 推荐落地顺序

1. 保留当前页面视觉与斥力参数不动
2. 把鼠标输入抽象成统一交互源
3. 接入前视 MediaPipe，先完成二维手部驱动
4. 在投影现场验证识别稳定性与交互节奏
5. 再加手机侧拍、墙面线标定和深度阈值

## 当前设计结论

- 现在最值得做的是“前视单机 MVP”
- 手机侧拍不是当前第一步，但它是后续实现真实空间接近的必要条件
- 不建议一开始就做复杂手势识别，当前需求只需要“手在何处、是否靠近”
