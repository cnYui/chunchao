# 2026-04-28 最终合成器手工布局编辑器设计

## 背景

- 当前最终合成器页的 `16` 格 ROI 和 `VIBE / VOL / REV / POS` 控件区域，主要依赖“页面 DOM 控件位置 -> 四点标定透视映射 -> 摄像头采样区域”这条链路生成
- 该方案的问题不是检测算法本身，而是几何来源不可靠：现场机位、投影角度和控制台摆位一旦偏移，自动推导出的区域就会和真实画面错位
- 用户已明确要求改为“自己在右下预览里手工画区域，再按画出来的区域识别和调试”

## 本次目标

- 在最终合成器页新增“手工布局绘制模式”
- 允许用户直接在右下摄像头预览里按顺序绘制：
  - `16` 个 pad 格子
  - `VIBE`
  - `VOL`
  - `REV`
  - `POS`
  - `控制台外框`
- 绘制结果保存为独立布局数据，后续 ROI 检测、手势控制和调试描边统一读取该数据
- 保留旧自动几何作为兜底，但只有在没有手工布局时才启用

## 已确认范围

- 第一版只支持矩形拖拽绘制，不支持四边形、多边形或自由曲线
- 第一版只支持固定顺序绘制，不支持跳步
- 第一版支持：
  - `撤销上一个`
  - `重画当前`
  - `清空全部`
  - `保存布局`
  - `退出布局`
- 第一版不支持：
  - 已完成矩形的边角拖拽微调
  - 多套布局切换
  - 服务端保存
  - 控制台外框参与识别

## 选型结论

### 方案对比

1. 手工矩形绘制模式
- 直接在预览里按真实画面拖出矩形
- 后续运行时直接消费这份布局

2. 自动映射基础上的手动微调
- 先用现有自动几何生成，再允许逐个修位置

3. 完全自由四边形绘制
- 每个区域四点单独标定

### 采用方案

- 采用方案 `1`

### 原因

- 当前问题的根因是“自动生成几何不可信”，不是“还缺一层微调”
- 用户需要的是现场可重复操作的所见即所得流程，而不是先猜再修
- 第一版只要矩形即可覆盖 pad ROI 和控件命中需求，没必要把编辑器复杂度抬高

## 交互流程

### 模式切换

- 右下预览分为：
  - `运行模式`
  - `布局模式`
- 进入布局模式后：
  - 暂停占格检测和手势控件映射
  - 禁止布局编辑以外的运行态交互
- 保存成功后回到运行模式

### 绘制顺序

1. `Pad 1` 到 `Pad 16`
2. `VIBE`
3. `VOL`
4. `REV`
5. `POS`
6. `控制台外框`

### 绘制行为

- 鼠标按下开始绘制
- 拖动显示当前矩形
- 鼠标松开完成当前项
- 完成后自动进入下一项

### 辅助信息

- 顶部显示当前待绘制项，例如 `请绘制 Pad 7`
- 同时显示进度，例如 `6 / 21`
- 已完成区域持续显示名称和描边
- 当前绘制项高亮显示

## UI 设计

### 按钮

- `布局模式`
- `撤销`
- `重画当前`
- `清空`
- `保存布局`
- `退出布局`

### 显示规则

- 布局模式：
  - 显示所有已完成框
  - 显示当前正在拖拽的预览框
  - 高亮当前目标项
- 运行模式：
  - 显示已保存布局描边
  - 显示被占用的 pad 高亮
  - 显示当前命中的控件区
- 旧自动推导的 ROI 调试框不再与手工布局同时显示，避免用户看到两套几何

## 数据模型

### 原始布局数据

- 坐标系使用右下预览内部坐标
- 保存结构：

```js
{
  version: 1,
  previewSize: { width, height },
  pads: [
    { id: 'pad-1', label: 'Pad 1', x, y, width, height },
    ...
    { id: 'pad-16', label: 'Pad 16', x, y, width, height }
  ],
  controls: {
    vibe: { id: 'vibe', label: 'VIBE', x, y, width, height },
    volume: { id: 'volume', label: 'VOL', x, y, width, height },
    reverb: { id: 'reverb', label: 'REV', x, y, width, height },
    position: { id: 'position', label: 'POS', x, y, width, height }
  },
  consoleFrame: { id: 'console-frame', label: '控制台', x, y, width, height }
}
```

### 存储策略

- 第一版存入 `localStorage`
- 存储键独立命名，例如 `synthManualLayout`
- 若后续要导入导出，再基于该结构扩展 JSON 序列化即可

### 运行态派生数据

- 根据当前预览元素尺寸，把原始布局按比例换算到实时视频采样坐标
- 派生出：
  - `padRois`
  - `controlRects`
  - `consoleFrameRect`

## 运行时接入

### 新状态源

- 新增 `manualLayout` 状态
- 新增 `layoutMode` 状态
- 新增“从原始布局派生运行态矩形”的纯函数模块

### ROI 检测

- `samplePadFeatures(...)` 不再默认读取自动映射生成的 `state.padRois`
- 运行时优先读取手工布局派生出的 `padRois`
- 仅当手工布局不存在时，才回退到旧自动几何

### 手势控制

- `applyHandControl(...)` 的命中区域改为优先读取手工布局里的：
  - `VIBE`
  - `VOL`
  - `REV`
  - `POS`
- 这样摄像头预览中看到的控制区与真实命中区保持一致

### 调试层

- 调试描边统一基于同一份手工布局派生数据
- 布局模式和运行模式只切换展示样式，不切换几何来源

### 旧四点标定的职责调整

- 四点标定继续保留
- 但职责降级为：
  - 初始化摄像头预览参考
  - 在没有手工布局时提供旧自动几何兜底
- 不再作为手工布局模式下 `16` 格和控制区的主要来源

## 错误处理

- 若用户未完成 `21` 个区域，不允许保存
- 若某个矩形宽高过小，则视为无效并要求重画
- 若 `localStorage` 中布局数据结构损坏，则忽略并回退到旧几何兜底
- 若预览尺寸变化，则重新派生运行态矩形，但不改动原始布局数据

## 测试策略

- 新增纯函数测试：
  - 布局步骤顺序
  - 原始布局校验
  - 预览坐标到运行态坐标的比例换算
- 新增页面结构测试：
  - 布局模式按钮存在
  - `21` 个步骤定义完整
- 回归测试：
  - 现有手势控制测试继续通过
  - 现有 ROI 检测测试继续通过

## 分阶段实现

1. 抽出布局步骤定义与布局数据结构
2. 在右下预览加入布局模式 UI 和拖拽绘制层
3. 接入 `localStorage` 持久化
4. 接入运行态派生矩形
5. 让 ROI 检测和手势命中优先使用手工布局
6. 更新调试显示与回退逻辑

## 风险与取舍

- 第一版不做已完成矩形的拖拽微调，原因是先把“可画、可存、可跑”做稳
- 继续保留旧自动几何作为兜底，原因是避免布局数据缺失时页面完全不可用
- 使用预览坐标系而不是页面 DOM 坐标，原因是用户实际看到和操作的就是预览画面

## 结论

- 本次重构的核心原则是：几何真相由用户手工定义，识别算法只消费这份真相，不再反过来生成真相
- 第一版实现目标不是做一个通用图形编辑器，而是做一个现场可重复使用的最小布局工具
