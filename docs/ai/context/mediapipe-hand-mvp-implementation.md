# MediaPipe 手部输入 MVP 实现记录

## 本次改动目标

- 保留当前排练室场景与乐器斥力表现层
- 把输入从“鼠标专用”改成“统一交互源”
- 接入浏览器端 MediaPipe Hand Landmarker
- 左手和右手都可驱动交互
- 鼠标保留为后备输入，避免摄像头或模型初始化失败后页面失效

## 当前实现方式

- 仍然使用静态页面，不引入前端框架
- `main.js` 里新增两类输入源：
  - 鼠标输入
  - 手部输入
- 场景响应层只消费统一输入源，不直接依赖某一种输入设备
- 页面右下角增加了一个小型摄像头预览窗口，并在视频上叠加手部关节点与骨架

原因：

- 现场调试时需要直接确认摄像头有没有识别到手和关节
- 不额外引入说明文字，也能快速判断识别状态

## 手部输入策略

- 使用 MediaPipe Hand Landmarker 的 `VIDEO` 模式
- 同时允许识别两只手
- 每一帧只选择一只“主交互手”
- 主交互手优先延续上一帧的左右手身份，避免在两只手之间来回跳

## 交互点选择

- 当前使用掌心近似中心作为接近源
- 计算点来自以下关键点平均值：
  - wrist
  - index_mcp
  - middle_mcp
  - ring_mcp
  - pinky_mcp

原因：

- 比指尖更稳定
- 更适合“靠近乐器就斥开”的接近式交互

## 平滑与性能

- 对掌心坐标做了轻量平滑，先抑制投影环境下的抖动
- 手部识别帧率限制在约 `30fps`
- 当前不引入更重的滤波器，先保持实现简单

## 运行前提

- 摄像头访问要求安全上下文
- 也就是需要在 `localhost` 或 `https` 下打开页面
- 如果直接双击本地 `html` 走 `file://`，手部输入不会启动

## 当前依赖取舍

- 由于项目没有构建链路，本次先通过官方 CDN 动态加载 `@mediapipe/tasks-vision`
- 手部模型也先使用官方托管地址
- 这是对“尽量不新增外部依赖”的一次受控例外

原因：

- 当前需求本身就是引入 MediaPipe 能力
- 先验证装置交互比先搭建工程化链路更重要

## 当前局限

- 现在只解决“二维画面内接近”
- 还没有解决“手真的贴近墙面”这个深度问题
- 当前没有提供可视化校准 UI
- 摄像头画面与投影区域默认按整幅画面映射，若现场机位有透视偏差，后续需要增加标定参数

## 下一步

1. 在 `localhost` 环境下验证前视手部识别稳定性
2. 根据现场机位决定是否需要增加 `x / y` 映射裁剪参数
3. 第二阶段接入侧拍手机距离通道，给斥力增加真实深度阈值
