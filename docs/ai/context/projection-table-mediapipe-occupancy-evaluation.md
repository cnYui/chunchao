# 最终展示页桌面投影交互评估

## 评估结论

- 方案可以做，但需要拆成两个输入通道：
  - 手指调控：使用 MediaPipe Hand Landmarker 识别手部 21 个关键点，再映射到页面坐标
  - 方块占格：不建议用 MediaPipe Hand Landmarker 判断，应该用桌面摄像头画面里的 16 个 ROI 做遮挡 / 占用检测
- MediaPipe Hand Landmarker 适合检测手、手指、左右手和关键点，不负责识别桌面上的实体方块
- 若只用电脑自带前置摄像头，通常无法稳定判断桌面 16 个格子是否被方块遮挡；需要俯拍或斜俯拍摄像头能看到投影桌面
- 如果要把摄像头画面坐标稳定映射到投影页面坐标，需要做四点标定并计算单应性变换

## 推荐架构

1. 页面进入最终展示页后启动摄像头。
2. 标定投影页面四角或控制台四角，得到 camera -> page 的坐标变换。
3. 对 16 个 `.pad-btn` 读取 `getBoundingClientRect()`，得到页面里的格子区域。
4. 把每个格子的页面区域反算到摄像头画面中，形成 16 个检测 ROI。
5. 空场状态采集一帧或多帧作为 baseline。
6. 每帧对 16 个 ROI 做亮度差、颜色差、纹理变化和边缘变化检测。
7. 对检测结果做 hysteresis / debounce，只有从未遮挡变成遮挡时触发 `playTone(freqs[i])`。
8. 手指调控继续使用 MediaPipe 手部关键点，例如食指指尖控制旋钮、滑杆或其他功能。

## 为什么不直接用 MediaPipe 判断方块

- Hand Landmarker 的输出是手部 landmarks，不是通用遮挡或物体占用结果。
- Object Detector 可以检测物体，但需要对应模型；通用 COCO 模型不一定有这些实体方块类别。
- 对固定 4x4 网格，传统 ROI 遮挡检测更简单、更快、更可控，不需要训练模型。

## 关键风险

- 投影内容本身会变化，遮挡检测必须先做空场 baseline，不能只用固定阈值。
- 用户的手经过格子也会造成遮挡，需要区分“手经过”和“实体方块放下”。建议用持续时间阈值，例如遮挡稳定超过 300-500ms 才认定为方块。
- 环境光变化会影响 ROI 差分，需要做自动归一化或周期性更新 baseline。
- 方块如果透明、反光或颜色接近投影内容，会降低可靠性。
- 摄像头视角越斜，格子形变越明显，越依赖标定质量。

## MVP 建议

- 第一版只做 16 格占用检测，不先做方块类别识别。
- 方块触发逻辑先采用 `empty -> occupied` 的边沿触发，避免一个方块长期放在格子上导致声音连续重触发。
- 手部调控和方块占格可以共享同一个摄像头，但逻辑上分开：MediaPipe 只服务手，ROI 差分服务方块。
- 若现场需要高稳定性，建议使用独立俯拍摄像头，而不是笔记本前置摄像头。

## 资料依据

- Google MediaPipe Web Hand Landmarker 文档说明，Web 端可用 `@mediapipe/tasks-vision`，视频模式使用 `detectForVideo()` 逐帧处理摄像头画面。
- Hand Landmarker 输出手部 landmarks、world landmarks 和左右手信息，适合做手指位置映射。
- Google 文档也说明 `detect()` / `detectForVideo()` 是同步调用，摄像头逐帧检测可能阻塞主线程，后续正式实现建议放入 Web Worker 或降低检测频率。
