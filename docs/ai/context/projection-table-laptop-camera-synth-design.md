# 桌面投影合成器交互设计

## 文档目的

本文定义 `synthesizer.html` 在桌面投影场景下的新交互设计。
目标是让用户通过电脑内置摄像头，在同一个页面里同时完成两类输入：

- 手指调控旋钮和滑杆
- 将实体方块放到 16 个格子上触发对应声音

本文只定义设计，不进入实现细节代码。

## 已确认前提

- 现场必须只使用电脑自带摄像头，不增加俯拍相机或手机侧拍通道
- 电脑会被倾斜摆放，内置摄像头能够稳定拍到整张被投影的桌面区域
- 16 个格子中任意格被方块占住时，该格开始发声；方块移开后停止发声
- 第一版不识别“是哪一种方块”，只识别“这个格子是否被占用”
- 页面仍以现有 `synthesizer.html` 为最终展示页，不另起新交互页面

## 问题拆分

当前需求表面上是“调用 MediaPipe 做方块占格和手指调控”，但这其实包含两种不同性质的视觉任务：

- 手指调控：属于手部关键点追踪问题
- 格子占格：属于固定区域的遮挡 / 占用检测问题

这两类问题不能强行用一个模型统一处理。
`MediaPipe Hand Landmarker` 适合做手部关键点，不适合直接判断桌面方块是否占格。

因此系统必须拆成两路输入：

- `Hand Tracker`
- `Occupancy Detector`

## 方案比较

### 方案 A：ROI 占格检测 + MediaPipe 手部调控

- 使用 `MediaPipe Hand Landmarker` 处理手部输入
- 使用 16 个固定 ROI 的差分 / 边缘 / 纹理变化处理方块占格

优点：

- 不需要训练模型
- 不需要改实体方块
- 与当前 4x4 格子页面结构天然匹配
- 第一版可在静态页面基础上直接接入

缺点：

- 对环境光、桌面材质、投影亮度敏感
- 必须做标定、baseline 和防抖
- 只用内置摄像头时稳定性上限有限

### 方案 B：ROI 占格检测 + 标记辅助

- 保留方案 A 的总体结构
- 在桌面或方块上增加低干扰标记，提高占格判定稳定性

优点：

- 稳定性更高
- 后续可以进一步扩展为方块类别识别

缺点：

- 需要改现场物料
- 不符合当前“纯现有条件先跑通”的目标

### 方案 C：单一视觉模型同时识别手和方块

- 用 MediaPipe 或通用检测模型统一处理手和方块

优点：

- 概念上统一

缺点：

- 选型错误
- `Hand Landmarker` 不负责方块占格
- 通用模型对自定义实体方块不稳定
- 成本高，且第一版没有必要

## 选型结论

第一版采用 `方案 A`。

原因：

- 满足“只用内置摄像头”的硬约束
- 不要求修改实体方块
- 能先做出可运行的 MVP
- 后续如果现场误判压不住，再升级到 `方案 B`

## 系统边界

### 输入边界

- `Hand Tracker` 只负责手部关键点、左右手、手势和控件调节输入
- `Occupancy Detector` 只负责 16 个格子的占用状态
- 两路输入不能直接互相覆盖结果，必须通过统一路由层汇总

### 声音边界

每个格子的声音语义固定为：

- `empty -> occupied`：开始发声
- `occupied -> empty`：停止发声

因此当前“按钮点击一次只播放一个短音”的逻辑不再足够。
页面后续需要抽象出可持续控制的声音生命周期接口，例如：

- `startPadVoice(id)`
- `stopPadVoice(id)`

### 非目标

第一版明确不做以下内容：

- 不识别具体方块类别
- 不判断方块朝向
- 不解决真实三维深度距离
- 不追求完全免调试的现场即插即用

## 坐标标定设计

### 标定目标

由于电脑会倾斜放置，摄像头看到的桌面画面一定存在透视形变。
因此不能直接用页面像素坐标判断方块是否占格，必须建立：

- `camera space`
- `page space`

之间的稳定映射关系。

### 第一版标定方式

第一版采用手动四点标定。

流程：

1. 页面显示投影控制台的 4 个明显角点
2. 摄像头预览窗口显示当前桌面画面
3. 用户在摄像头画面中依次点击对应的 4 个角点
4. 系统根据 4 对点计算单应性变换
5. 后续所有格子 ROI 都通过这组变换进行 `page -> camera` 映射

选择手动四点标定的原因是：

- 内置摄像头视角不稳定
- 投影亮度和桌面纹理会影响自动检测
- 第一版优先可靠，不优先全自动

## 格子占格检测设计

### ROI 生成

标定完成后：

1. 读取 16 个 `.pad-btn` 的 `getBoundingClientRect()`
2. 得到每个格子在页面中的矩形区域
3. 将矩形四角映射到摄像头画面中
4. 得到 16 个透视四边形 ROI

每个 ROI 绑定一个独立状态机。

### baseline 采集

占格检测必须基于空场参考，不能只靠固定阈值。

流程：

1. 标定完成后要求桌面 16 格为空
2. 连续采样若干帧
3. 为每个 ROI 生成 baseline 特征

baseline 建议至少保存：

- 平均亮度
- 亮度方差
- 边缘密度
- 粗粒度颜色分布

原因：

- 投影画面本身带有纹理和亮暗变化
- 单纯像素差分过于脆弱

### 每帧检测

每一帧对每个 ROI 计算 `occupancyScore`，分数来自以下组合：

- 当前帧与 baseline 的亮度差
- 当前帧与 baseline 的边缘差
- 当前帧与 baseline 的局部纹理变化
- ROI 内稳定遮挡面积比例

然后采用双阈值判定：

- 高于 `enterThreshold` 且持续超过阈值时间：认定进入占格
- 低于 `exitThreshold` 且持续超过阈值时间：认定退出占格

不使用单阈值的原因是要减少抖动。

### 占格状态机

每个格子不只保存一个布尔值，而是使用四态状态机：

- `empty`
- `candidateOccupied`
- `occupied`
- `candidateEmpty`

这样可以吸收短时误检，避免手经过或亮度闪动导致来回跳变。

## 手部调控设计

### 手部输入职责

`Hand Tracker` 只输出：

- 左右手信息
- 21 个关键点
- 关键控制点位置，例如食指指尖、掌心中心
- 可能的控制状态，例如是否进入某个控件区域

### 第一版手部调控范围

第一版只允许手部调控现有连续型控件：

- `VIBE` 旋钮
- `VOL`
- `REV`
- `POS`

16 个 pad 区域不接受“手经过即触发声音”，真正触发仍由实体方块完成。

原因：

- 当前目标是做“方块占格 + 手部调控”共存
- 不是把整个控制台改造成纯手势乐器

## 手部与占格共存设计

### 输入隔离

系统逻辑上拆成三层：

- `Hand Tracker`
- `Occupancy Detector`
- `Interaction Router`

其中：

- `Hand Tracker` 不直接控制音频
- `Occupancy Detector` 不直接操作 DOM
- 统一由 `Interaction Router` 把输入转成页面和声音状态

### 误判抑制

为避免手经过格子被误认成“方块放下”，第一版必须同时使用三层过滤：

- 时间过滤  
  遮挡持续超过 `300-500ms` 才允许进入 `occupied`

- 手部区域排除  
  根据手部关键点生成粗略 `handMaskRegion`，若某 ROI 的变化主要来自该区域，则延迟认定

- 状态机防抖  
  不允许一帧内直接从 `empty` 跳到 `occupied`

### 冲突规则

以下规则需要固定写入实现：

- 手调旋钮或滑杆时，不暂停格子占格检测
- 若某格子的变化主要来自手部区域，不触发 `occupied`
- 若格子已 `occupied`，手经过其上方不能触发 `empty`
- 只有持续满足“无占格”条件时，才允许 `occupied -> empty`

## 模块设计

第一版建议按以下模块划分：

- `camera-controller`  
  管理摄像头启动、权限、视频流和帧获取

- `projection-calibration`  
  管理四点标定和坐标变换

- `occupancy-detector`  
  管理 16 格 ROI、baseline、占格分数和状态机

- `hand-controller`  
  管理 MediaPipe Hand Landmarker 和手部控制输入

- `synth-router`  
  管理两路输入到页面控件和音频系统的映射

拆模块的原因是：

- 摄像头、MediaPipe、CV 检测、页面控件、音频状态的变化频率不同
- 如果全部写进同一脚本，后续调阈值和排查现场问题会非常困难

## 页面接入方式

接入目标固定为 `synthesizer.html`，不另起新页面。

推荐接入顺序：

1. 页面先正常显示现有合成器 UI
2. 用户授权摄像头
3. 启动摄像头预览
4. 进入四点标定模式
5. 标定成功后采集 baseline
6. baseline 完成后进入正式运行态
7. 正式运行态下并行执行：
   - 手部检测
   - 占格检测
   - 声音状态同步

## 调试层设计

第一版必须保留一个轻量调试层，仅在调试模式打开：

- 摄像头预览小窗
- 标定四点
- 16 个 ROI 边框
- 每个格子的占格状态
- baseline 是否已采集完成

保留调试层的原因是：

- 这套系统最大的风险来自现场误判
- 没有调试层就无法快速判断问题来自标定、阈值、光照还是摄像头角度

## 性能策略

第一版性能策略如下：

- 摄像头分辨率优先控制在 `640x360` 或 `960x540`
- MediaPipe 手部检测控制在大约 `15-20 FPS`
- ROI 占格检测只处理 16 个小区域，不做整帧重型图像处理
- 若页面交互出现明显卡顿，再考虑将占格检测移入 `Web Worker`

这里要明确：

- MediaPipe Web 逐帧检测本身就有开销
- 在只用内置摄像头和浏览器主线程的前提下，第一版不能再叠加重型 CV 模型

## 验证标准

第一版通过标准定义为：

- 用户能在一次标定流程内进入正式运行态
- 任一格放入方块后，`300-500ms` 内开始发声
- 方块移开后，`300-500ms` 内停止发声
- 手经过格子上方时，不应频繁误触发占格
- 手部可以稳定调节 `VIBE`、`VOL`、`REV`、`POS`
- 最终展示页能在投影现场连续运行一段时间，不出现大面积状态漂移

## 风险与结论

本方案在“只用电脑内置摄像头”的约束下，可以作为 MVP 落地。

但必须明确：

- 稳定性上限会受到摄像头角度、桌面材质、环境光和用户动作影响
- 第一版的目标应当是“可运行、可调参、可现场校准”
- 不应把第一版目标定义成“完全免调试、完全零误触”

后续若现场误触率压不住，升级方向优先级如下：

1. 优化标定、阈值和调试层
2. 引入低干扰标记辅助占格检测
3. 最后才考虑更重的视觉模型
