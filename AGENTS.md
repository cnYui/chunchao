# 项目记忆

## 当前前端方向

- 当前项目以静态浏览器场景为起点，不先引入框架
- 原因：现阶段只有图片素材，目标是尽快验证交互概念

## 已确认的交互目标

- 使用 `pic/` 下的背景图和乐器贴图构建排练室场景
- 用户鼠标接近乐器时触发斥力效果
- 用户鼠标移开后，乐器恢复到原始位置
- 当前新的装置目标是把“鼠标接近”升级为“真实手部接近”
- 左手或右手都可以作为交互输入
- 若目标是“手真的靠近墙上的乐器才触发”，后续必须补充侧拍手机通道做距离估算

## 当前实现约束

- 优先复用现有素材，不新增外部依赖
- 先解决核心交互，再考虑音效、剧情或更复杂编排
- 如果后续需要更复杂的时间线、音频同步或多场景切换，再评估是否升级为正式前端工程

## 交互方案记忆

- 当前把用户提到的 `Media Pipeline` 按 `MediaPipe` 理解
- 第一阶段优先做前视单机 MVP：电脑摄像头 + MediaPipe Hand Landmarker，把手部二维位置映射成乐器斥力输入
- 第一阶段先解决“画面内接近”，不强行解决真实墙面深度
- MediaPipe 的手部 `z` 和 `world landmarks` 不能直接当作“手到墙面距离”
- 第二阶段若要实现真实空间接近，使用侧拍手机估算手到墙面的距离，并通过局域网把数据回传到投影电脑页面
- 第二阶段若启用手机侧拍通道，需要引入最小通信层，不能假设纯静态页面自己充当 WebSocket 服务端
- 第一阶段交互点优先使用掌心中心，不优先使用指尖，原因是掌心抖动更小、更适合作为接近源
- 第一阶段已在 `main.js` 中把输入层抽象为统一交互源，鼠标保留为后备输入
- 第一阶段当前通过官方 CDN 动态加载 `@mediapipe/tasks-vision`，原因是项目仍保持静态页形态、先验证交互
